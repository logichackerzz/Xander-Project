## GSAP 是什麼，為何選它

### GSAP 是什麼

**GSAP（GreenSock Animation Platform）** 是一套 JavaScript 動畫函式庫，核心概念是「把任何 CSS 屬性、DOM 屬性、甚至 JavaScript 物件的數值，從 A 平滑過渡到 B」。

```js
// 基本用法：元素在 1 秒內從 y:0 移動到 y:100，ease 是緩動曲線
gsap.to('.box', { y: 100, duration: 1, ease: 'expo.out' })

// Timeline：把多個動畫排成序列，或部分重疊
const tl = gsap.timeline()
tl.to('.a', { opacity: 1 })
  .to('.b', { x: 50 }, '-=0.3')  // 提前 0.3s 開始
```

GSAP 本身不依賴任何框架（不是 React 專用），可以直接操作 DOM，非常適合 Vanilla JS 或 Vite 這類輕量環境。

### ScrollTrigger（GSAP 插件）

ScrollTrigger 是 GSAP 官方插件，核心功能：
- **Pin（釘住）**：滾動到某個元素時，把它「凍結」在視口，畫面停在那裡，使用者繼續滾動才放開
- **OnEnter / OnLeave**：滾動觸發事件，用來控制動畫播放
- **Scrub**：把滾動進度直接對應到動畫進度（像 Apple 的拆解動畫）

本次投影片用的是 Pin + OnEnter 模式：每個 section 釘住一個視口高度，滾入時播放進場動畫。

### Lenis（平滑滾動）

Lenis 是一套用 `requestAnimationFrame` 實作的平滑滾動函式庫，讓滾動有「慣性」感覺（像 macOS 原生的 momentum scroll）。透過 GSAP ticker 連結，讓兩套系統同步：

```js
const lenis = new Lenis({ lerp: 0.08, smoothWheel: true })
lenis.on('scroll', ScrollTrigger.update)
gsap.ticker.add((time) => lenis.raf(time * 1000))
```

`lerp: 0.08` = 每幀移動到目標位置的 8%，數字越小越飄。

## 為何選 GSAP，不選其他方案

| 方案 | 適合場景 | 為何沒選 |
|------|---------|---------|
| **CSS Animation** | 簡單重複動畫 | 無法精確控制 timeline、無法 scroll trigger |
| **Framer Motion** | React 元件動畫 | 綁定 React，投影片想用輕量 Vanilla 架構 |
| **AOS / animate.css** | 快速上 scroll 動畫 | 只有 class toggle，無法做 timeline 排序 |
| **Three.js / WebGL** | 3D / particle | 學習成本高，投影片不需要 |
| **open-slide.dev** | React 投影片框架 | 主打純文字，動畫能力弱 |
| **Reveal.js / impress.js** | 傳統投影片 | 鍵盤翻頁模式，不是捲軸體驗 |

**選 GSAP 的三個核心理由：**
1. **Timeline 控制精準** — 可以定義「A 結束後 0.3s，B 開始，C 在 B 還沒結束時插入」，這種精度 CSS 做不到
2. **ScrollTrigger 是業界標準** — 大型行銷網站（Apple、Stripe、gsap.com 自己）都用這套做滾動體驗
3. **無框架依賴** — 投影片是純 HTML + Vite 的 Vanilla JS 架構，GSAP 直接操作 DOM 完全無問題

## 關鍵技術決策紀錄

### 為何不用 char-level Splitting 做 Hero

原本計劃把「Folio」拆成 5 個 `.char` span，逐字落入。但發現：
- `-webkit-background-clip: text` 的漸層效果只能套在有背景的那個元素本身，拆成子 span 後漸層效果消失
- Splitting 需要 `data-splitting` attribute，沒加就是空陣列，動畫靜默失效

**最終決策**：Hero Title 直接用 `yPercent: 115 → 0`，搭配 `.hero-title-wrap { overflow: hidden }`，整個字從底部滑入，漸層完整保留。效果一樣有衝擊力。

### 為何 Closing 用 Typewriter 而非 Scramble

Scramble 函數用 ASCII 亂碼字元（A-Z, 0-9, █▓▒）替換中文字符。因為 ASCII 是單寬字元、中文是雙寬字元，替換時文字寬度會跳動，造成視覺閃爍。

**最終決策**：自製 `typewriter()` 函數，逐字追加 `el.textContent`，配合中文字元寬度一致，視覺效果乾淨。
