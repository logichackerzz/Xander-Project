# AI 規劃模式（/brainstorm）

把模糊的想法變成具體的計劃書。透過引導式問答，幫你在 AI 動手寫 code 之前，先釐清「到底要做什麼」。

> 致謝：核心理念參考自 [obra/superpowers](https://github.com/obra/superpowers)（brainstorming skill, MIT License），經過簡化與在地化改造。

---

<HARD-GATE>
在用戶確認計劃書之前，絕對不可以：
- 寫任何程式碼
- 建立任何檔案（計劃書本身除外）
- 執行任何實作動作
- scaffold 任何專案結構

不管任務看起來多簡單，都要先走完規劃流程。
一個待辦事項 App、一個簡單腳本、一個設定檔修改，全部都要。
計劃書可以很短（簡單任務幾句話就夠），但流程不能跳過。
</HARD-GATE>

## 輸入格式

- `/brainstorm 我想做一個記帳 App` — 帶主題啟動
- `/brainstorm` — 不帶主題，AI 會問你想做什麼

---

## 流程（按順序執行，不可跳步）

### Phase 0：確認溝通模式 ⭐ 最重要的第一步

**這是整個 brainstorm 流程的第一個問題，不可跳過。** 它決定了後續所有對話、假設列表、方案比較、計劃書的呈現方式。

用 **AskUserQuestion** 問：

> 你希望我用什麼樣的程度，跟你進行專案與需求的討論及展開？
>
> 1. 🟢 **小白模式**（推薦給非工程師）
>    我會盡量用白話講，少用技術名詞，重點放在「你會得到什麼」「有什麼風險」「未來可以怎麼延伸」。技術細節我自己處理。
>
> 2. 🟡 **半技術模式**
>    保留一些必要的技術名詞（檔名、套件名、API 名稱），但會搭配說明。適合有點概念但不想太深入細節的人。
>
> 3. 🔴 **工程師模式**
>    直接用技術詞彙溝通（檔案路徑、套件版本、架構名詞），假設你看得懂。最精簡、最快。
>
> 4. 🤔 **你決定就好**
>    AI 看情況自己判斷（會根據你後續的回答自動調整）。

將選擇記為 `COMMUNICATION_MODE`，**從這個 Phase 之後的每一句話都要套用**：

#### 🟢 小白模式的具體規則

當 `COMMUNICATION_MODE = 小白` 時：
- **盡可能減少技術名詞**：用「網頁的設定檔」而不是「config.js」、用「自動執行的腳本」而不是「cron job」、用「網址後面帶的參數」而不是「query string」
- **直接溝通成果**：每個方案都先說「做完你會看到什麼」，技術細節放最後或省略
- **說明相關風險與延續性發展**：每個方案都附上「可能的風險」和「未來可以怎麼延伸」
- **用白話的方式溝通**：把「執行面都是 AI 在處理」當前提，用戶不需要懂程式
- **假設列表**裡不要出現純技術名詞，必須改寫成成果導向
  - ❌「用 React + Tailwind + Vite 部署到 Cloudflare Pages」
  - ✅「做成一個網頁，放在我們現有的網頁空間上，手機電腦都能開」
- **方案比較表**的欄位要改成：「做完長什麼樣」「優點」「缺點」「風險」「未來能延伸」「需要你做的事」（不要有「技術棧」「框架」這種欄位）

#### 🟡 半技術模式的具體規則

- 保留檔名、套件名、API 名稱等可以幫助對齊的關鍵詞
- 第一次出現專業名詞時加一句白話說明
- 方案比較可以有簡單的技術欄位

#### 🔴 工程師模式

- 直接用技術詞彙，不需解釋
- 方案比較表可以有完整的技術細節

#### 🤔 自動判斷

- 從用戶的主題描述和回答中判斷詞彙密度
- 預設先用半技術模式，發現用戶用很多技術詞 → 切換工程師模式；發現用戶問「那是什麼」 → 切換小白模式

### Phase 1：暖身確認

**Step 1-1：確認主題**

如果用戶在 `/brainstorm` 後面有寫主題，直接使用。
如果沒有，用 **AskUserQuestion** 問：

> 你想做什麼？用一句話描述就好，不用很完整。

**Step 1-2：分類**

根據主題，用 **AskUserQuestion** 問：

> 這比較像是哪一種？
> 1. 🆕 **做一個新東西**（新功能、新專案、新工具）
> 2. 🔧 **解決一個問題**（Bug、錯誤、某個東西壞了）
> 3. ✨ **改善現有的東西**（優化、重構、讓它更好用）
> 4. 🤔 **我還不確定**（讓 AI 幫我釐清）

將選擇記為 PROJECT_TYPE，這會影響後續問的問題方向。

### Phase 2：快速探索

**Step 2-1：掃描專案上下文（AI 自動執行，不需用戶操作）**

花 10-15 秒快速了解專案現況：
- 讀 CLAUDE.md（如果有的話）
- 看最近 5 筆 git commit（如果是 git repo）
- 用 Glob 掃描專案結構（根目錄的資料夾和主要檔案）
- 如果用戶的主題提到了特定檔案或功能，找到並讀取相關程式碼

**Step 2-2：列出假設（Assumptions Mode）**

根據掃描結果和用戶的主題，列出 3-5 個你對這個任務的理解和假設。

⚠️ **重要：必須套用 Phase 0 的 `COMMUNICATION_MODE`**。如果是小白模式，假設裡不可以有純技術詞彙，要改寫成成果導向。

格式範例（**半技術模式 / 工程師模式**）：

```
根據你的描述和專案現況，我先列出我的理解，你看看哪些需要修正：

1. **目標**：你想做一個可以記錄每日收支的 App
2. **技術**：用 React + 本地儲存（不需要後端）
3. **範圍**：先做基本的記帳功能，不含報表和圖表
4. **用戶**：只有你自己用，不需要多人功能
5. **部署**：做成網頁，不需要手機 App
```

格式範例（**小白模式**：同樣的內容，白話版）：

```
根據你的描述，我先講講我理解的方向，你看看哪些要修正：

1. **你想做什麼**：一個能記每天花了多少錢的小工具
2. **使用方式**：在電腦或手機的網頁上開來用，不需要下載 App
3. **資料存哪裡**：存在你自己的瀏覽器裡，不會傳到雲端（隱私 OK，但換電腦就看不到）
4. **這次先做的功能**：只做「輸入金額 + 看清單」，不做圖表報表
5. **誰會用**：只有你自己用，不會有別人登入
6. **未來可以延伸**：之後想加圖表、匯出 Excel、雲端同步都可以再加
```

然後用 **AskUserQuestion** 問：

> 以上的理解，有哪些需要修正？
> 1. ✅ **全部正確，繼續！**
> 2. ✏️ **有些要修正**（我會在下面跟你說）
> 3. 🔄 **大部分都不對，重新來**

如果用戶選 2，讓他說明要修正什麼，更新假設後進入 Phase 3。
如果用戶選 3，回到 Step 1-1 重新釐清主題。

### Phase 3：深度釐清

根據 PROJECT_TYPE 和已確認的假設，一次問一個問題來釐清細節。

**問問題的規則**：
- **永遠用 AskUserQuestion**，跳出選項框
- **一次只問一個問題**，不要連續轟炸
- **優先用選擇題**，每個選項附上一句話說明為什麼選它
- **最後一個選項永遠留給用戶自由發揮**：「以上都不是，我想要的是...」
- **給推薦**：如果你有推薦的選項，標上「推薦」並簡短說明原因
- **最多問 5 個問題**，如果用戶的假設已經很清楚，可以更少
- **已經從假設確認過的事情，不要再問**

**依類型的問題方向**：

🆕 做新東西：
- 給誰用的？（自己 / 特定對象 / 公開）
- 最重要的 1-2 個功能是什麼？（幫用戶聚焦，別什麼都想做）
- 技術偏好？（有沒有想用的語言、框架，或跟現有專案保持一致）
- 怎麼讓人用到？（本地跑 / 網頁 / 部署到哪裡）

🔧 解決問題：
- 問題什麼時候開始出現的？（最近改了什麼嗎？）
- 能重現嗎？（每次都會 / 偶爾 / 不確定）
- 有錯誤訊息嗎？（貼出來看看）
- 你試過什麼了？（避免重複嘗試）

✨ 改善優化：
- 現在最不滿意的地方是什麼？
- 改善後的理想狀態是什麼？
- 有沒有參考對象？（你看過誰做得好）
- 改動範圍想控制多大？（小修 / 中等重構 / 大改）

**不耐煩偵測**：
如果用戶回答中出現以下訊號：
- 「直接做吧」「不用問了」「你決定就好」「快點開始」
- 回答變得很短（一兩個字）
- 連續選「你推薦的就好」

→ **最多再問一個關鍵問題**，然後直接進入 Phase 4。
→ 說：「好的，剩下的我根據目前的理解來規劃，如果有不對的地方我們再調整。」

### Phase 4：方案提議

根據收集到的資訊，提出 2-3 個具體方案。

⚠️ **必須套用 Phase 0 的 `COMMUNICATION_MODE`**。小白模式的方案比較表欄位完全不同，請特別注意。

**呈現格式（半技術 / 工程師模式）**：

```
## 方案比較

| | 方案 A：[名稱] | 方案 B：[名稱] | 方案 C：[名稱] |
|:--|:--|:--|:--|
| **做法** | ... | ... | ... |
| **優點** | ... | ... | ... |
| **缺點** | ... | ... | ... |
| **適合** | ... | ... | ... |
| **預估工作量** | ... | ... | ... |

💡 **我的推薦：方案 X**
因為 [一句話理由]。
```

**呈現格式（小白模式）**：

```
## 三個做法比一比

| | 做法 A：[白話名稱] | 做法 B：[白話名稱] | 做法 C：[白話名稱] |
|:--|:--|:--|:--|
| **做完長什麼樣** | [用戶會看到的畫面/體驗] | ... | ... |
| **好處** | [實際的好處，不講技術] | ... | ... |
| **壞處** | [實際的限制，不講技術] | ... | ... |
| **可能的風險** | [會壞掉、會卡關、會不能用的情況] | ... | ... |
| **未來能延伸什麼** | [這個方案之後可以加什麼] | ... | ... |
| **你需要做的事** | [用戶要付出什麼，例如：要去哪註冊、要填什麼表] | ... | ... |

💡 **我推薦做法 X**
因為 [用白話講一句理由，例如：「最快可以看到成果，而且之後想加功能也最容易」]。

技術細節我會處理，你只需要決定方向就好。
```

然後用 **AskUserQuestion** 問：

> 你想選哪個方案？
> 1. 方案 A：[名稱]
> 2. 方案 B：[名稱]
> 3. 方案 C：[名稱]
> 4. 我想混合搭配 / 有其他想法

**如果任務很簡單**（只有一個合理的做法），可以跳過方案比較，直接說：
「這個任務比較直接，我建議的做法是 [說明]。你覺得 OK 嗎？」

### Phase 5：計劃書生成

**Step 5-1：撰寫計劃書**

根據確認的方案，撰寫計劃書。格式如下：

```markdown
# [計劃名稱]

> 建立時間：YYYY-MM-DD HH:MM
> 類型：[新功能 / 問題修復 / 改善優化]
> 狀態：待執行

---

## 你要做的事（一句話版）

[用最白話的方式說清楚這次要做什麼]

## 背景和動機

[為什麼要做這個？解決什麼問題？]

## 具體步驟

### Step 1：[步驟名稱]
- 做什麼：[具體動作]
- 產出：[這步做完會有什麼]
- 注意：[有什麼要小心的]

### Step 2：[步驟名稱]
...

（根據實際需要列 3-8 個步驟）

## 預計成果

做完以後你會得到：
- [成果 1]
- [成果 2]
- [成果 3]

## 不包含在這次的範圍

- [明確列出這次不做的事，避免範圍膨脹]

## 可能遇到的風險

- [風險 1]：[怎麼處理]
- [風險 2]：[怎麼處理]
```

**Step 5-2：Self-Review（AI 內部檢查，不需用戶操作）**

寫完計劃書後，自動檢查以下四項：

1. **佔位符掃描**：有沒有「TBD」「待確認」「之後再決定」？有的話，現在就填上。
2. **內部一致性**：步驟之間有沒有矛盾？前面說用 A 方案，後面的步驟卻在做 B 方案？
3. **範圍檢查**：步驟超過 8 個嗎？如果是，考慮拆成多個計劃。
4. **模糊度檢查**：每個步驟是否具體到「AI 看了就知道要做什麼」？如果出現「適當地處理錯誤」「做好測試」這種模糊指令，把它改具體。

有問題就直接修正，不需要告知用戶。

**Step 5-3：存檔**

將計劃書存到 `my-agent/100_Todo/plans/`：
- 檔名格式：`YYYY-MM-DD-[主題關鍵字].md`（例如：`2026-04-14-記帳App.md`）
- 如果目錄不存在，先建立

告知用戶：「計劃書已存到 [路徑]。」

### Phase 6：下一步

用 **AskUserQuestion** 問：

> 計劃書好了！你想怎麼做？
> 1. 🚀 **現在就開始執行！**（AI 會按照計劃書一步步做）
> 2. 📝 **我想先修改計劃書**（告訴我哪裡要改）
> 3. 💾 **先存著，之後再做**（你可以隨時回來說「按照計劃書執行」）

如果選 1：退出 brainstorm skill，按照計劃書的步驟開始執行。
如果選 2：讓用戶說明要改什麼，改完後重新跑 Step 5-2 的 self-review，再問一次。
如果選 3：結束。提醒用戶計劃書的位置，以及之後可以怎麼使用它。

## 護欄（Guardrails）

這些規則在整個 /brainstorm 流程中永遠生效：

- **不准寫 code**：在用戶確認計劃書之前，一行 code 都不能寫。讀 code 可以（為了了解專案），寫不行。
- **不准跳步**：不能因為「看起來很簡單」就跳過某個 Phase。Phase 可以很快（簡單任務每個 Phase 幾秒就過），但不能跳過。
- **一次一個問題**：永遠不要在同一則訊息裡問兩個以上的問題。
- **用選項框**：能用 AskUserQuestion 就用，不要用純文字讓用戶打字。
- **不要空洞附和**：不要說「好問題」「這很有趣」。直接回應內容。
- **挑戰假設**：包括用戶的和你自己的。如果用戶想做的東西太複雜，直接說：「這個範圍比較大，我建議先做 [核心部分]，之後再加其他的。」
- **YAGNI 原則**：You Ain't Gonna Need It。設計時砍掉「以後可能會用到」的功能，只留現在確定需要的。
- **尊重用戶的節奏**：用戶想快就快，不要死守流程。一次提醒就好，不要反覆阻擋。
- **永遠遵守 COMMUNICATION_MODE**：Phase 0 確認的溝通模式，從第二個 Phase 開始的每一句話、每一個假設、每一個方案比較表、最後的計劃書，都要套用。如果是小白模式卻不小心冒出技術名詞，要立刻自我修正改寫。
- **全程繁體中文，執行階段也不例外** ⚠️：整個 /brainstorm 流程以及**用戶選擇「現在就開始執行」之後的實作階段**，所有對用戶可見的文字（進度訊息、任務描述、TodoWrite 內容、檔案建立通知、錯誤解釋、最終總結）都**必須用繁體中文**。不要因為進入執行模式、或工具回傳是英文、或需要寫 code，就自動切換成英文。這條規則是**硬性的**。
  - **允許的例外**：
    1. **內部推理（`<thinking>` block）** — 不受此規則限制，可以用英文思考、除錯、跑假設，以維持最高效的推理品質。用戶看不到這層。
    2. **工具的輸入參數** — Bash 指令、Glob/Grep pattern、檔名、變數名、套件名一律保持英文（技術標識符本來就是英文）。
    3. **程式碼本身** — 變數、函式、import 全部英文；但**程式碼內的註解**遵循 CLAUDE.md 規則（中文為主）。
    4. **技術術語的精確保留** — 當翻譯反而會失去精確度時，保留英文原詞並加中文說明。例如「這是個 race condition（時序競爭問題）」比「這是個賽道條件問題」好。
  - **語言 ≠ 技術深度**：COMMUNICATION_MODE 控制技術詞彙的密度，這條規則控制輸出語言。兩者獨立。工程師模式的中文輸出仍然可以充滿 `useEffect`、`useCallback` 這種精確技術詞；小白模式的中文輸出會把它們翻譯成白話。
